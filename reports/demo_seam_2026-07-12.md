# Demo-Seam Audit — Summary + Artifacts result-card tabs

- **Date:** 2026-07-12
- **Scope (tight):** the newly added Summary + Artifacts tabs in `app/neuroad.html`
  (commit `bcca31c` "Add Summary + Artifacts tabs to result card, bound to live case data").
  Methods audited: `summaryEl`, `artifactsEl`, `discoveryFigure`, `_trailEl`/`_pipelineTrail`,
  `_reasoningBullets`, `_datasetRows`, and helpers (`_num`,`_ci`,`_aucStat`,`_confoundStat`,
  `_anchorStat`,`_valence`,`strengthDots`), plus `heroCase()`/`heroCohort()` (~1841–1843).
- **Method:** static code + data trace only. Browser E2E **not run** (another session owns the tab).
- **Source of truth:** `app/demo_data.json` (`meta.source:"engine"`, `deterministic:true`,
  `schema 1.1.0`). Live origin is `POST /api/investigate` → `result.case`; fallback is
  `substrates[substrate].cases.SURVIVOR/.KILL` + `.cohort`, plus top-level `discovery`.

## Verdict
The three data-value renderers driven by `heroCase()` (Summary chips, reasoning bullets,
dataset rows, pipeline trail) are **clean of hardcoded numeric literals** — every value traces
to a real case/cohort/test field. **But the Artifacts "Driving analysis" figure renders the
wrong discovery object** (planted-synthetic, not the real-cohort one) with its provenance
stripped, and there is a **visible outcome-AUC drift between the Summary tab and the Artifacts
tab** on the headline ADNI survivor. Details below.

---

## Findings (ranked by credibility risk)

### F1 — HIGH — Artifacts "Driving analysis" shows the *planted-synthetic* discovery, unbadged
- **Where:** `discoveryFigure(node)` `app/neuroad.html:2723` reads `data.discovery`; consumed as
  the Artifacts "Driving analysis" at `:2813`.
- **Issue:** `demo_data.json` has two discovery objects:
  - `discovery` — run on **planted ground truth** (`note:"…Recovery vs planted ground truth:
    ARI=1.0, AMI=1.0"`, three clean n=150 clusters named `tau_hot`/`age_atrophy`/`scanner_artifact`,
    **1 promotable** `tau_hot` @score 62, silhouette 0.063). This is the *method-validation on
    synthetic phantoms*, not a real-cohort result.
  - `discovery_real` — run on the **real cohort** (`substrate`,`real`,`embedding_dim`,`dx_mix`,
    n=96 across clusters 19/77, **0 promotable**, both "likely artifact (scanner/site)",
    no phenotype labels, silhouette 0.078).
  The new figure renders `discovery` (the synthetic one) under the label
  "Discovery clustering · unsupervised" / section "Driving analysis", inside a **real** ADNI/OASIS
  survivor's Artifacts, with a **promoted `tau hot` phenotype** and **no synthetic/planted badge**.
  This is the playbook's canonical A2/A3 violation: a synthetic/calibration result presented as a
  live driving analysis with its provenance stripped in render. The honest real-data version
  (`discovery_real`) tells the opposite story — 0 promotable, everything flagged as artifact.
- **Severity:** HIGH — a judge reads "we discovered a promotable tau-hot phenotype" as a real-cohort
  finding; the real run promotes nothing.
- **Minimal fix (owning session):** in `discoveryFigure`, either (a) read `data.discovery_real`
  for real-cohort cards, or (b) if `discovery` (planted) is shown deliberately as a method demo,
  add a "planted phantom / method validation" badge and relabel away from "Driving analysis" so it
  cannot read as this cohort's result. Do not silently keep the flattering synthetic figure under
  a real survivor.

### F2 — MEDIUM — Outcome-AUC drifts across tabs on the ADNI survivor (0.93 vs 0.92)
- **Where:** Summary AUC chip value = `naive_effect.value` via `_aucStat()` `:2645` (rendered `:2761`);
  Artifacts reasoning bullet uses `leakage_margin.outcome_auc` via `_reasoningBullets()` `:2701`.
- **Issue:** For ADNI SURVIVOR `naive_effect.value = 0.932` → Summary chip "AUC **0.93**", while
  `leakage_margin.outcome_auc = 0.922` → Artifacts bullet "Outcome AUC **0.92** beats the scanner-only
  classifier". Both are labeled "outcome AUC" to the user; toggling Summary↔Artifacts shows two
  different numbers for the same quantity. (OASIS survivor: both are 0.824, no drift — the drift is
  specifically on the score-100 headline ADNI hero.) The tree agrees with Summary (0.93, also from
  `naive_effect.value` @`_applyCaseToTree:1873`); only the Artifacts bullet diverges.
- **Severity:** MEDIUM — visible number inconsistency on the demo's headline case, same-label.
- **Minimal fix:** display one canonical outcome-AUC field in both tabs. Simplest: have the Summary
  chip source `leakage_margin.outcome_auc` (the same field the bullet + margin math use) so chip,
  bullet, and margin all agree; or reconcile the two fields in the data. (Field reconciliation is a
  backend call — flag only.)

### F3 — MEDIUM — AUC chip pairs a point estimate with a CI computed for a *different* value
- **Where:** `_aucStat()` `:2645` — `value:_num(ne.value)` (0.932) but `ci:_ci(lk.outcome_ci)`
  ([0.9061, 0.9383], the CI of `outcome_auc`=0.922). Rendered at `:2761`.
- **Issue:** The chip shows "AUC 0.93 [0.91–0.94]", but that CI is the confidence interval of
  `outcome_auc` (0.922), not of the displayed `naive_effect.value` (0.932). The interval happens to
  still contain 0.932, so it is not wrong on its face, but the value and its stated uncertainty come
  from two different quantities. Coupled with F2 this is the same root cause.
- **Severity:** MEDIUM (provenance mismatch; resolves if F2's canonical-field fix is applied).
- **Minimal fix:** show the CI belonging to the value displayed — i.e. pick `outcome_auc`+`outcome_ci`
  as the pair, or `naive_effect.value`+its own CI if one exists.

### F4 — LOW/MEDIUM — `_datasetRows` "REAL/SNAPSHOT" chip keys on *existence* of `co.badge`, not its meaning
- **Where:** `_datasetRows()` `:2713` `src:(co.badge?'real':'snapshot')`; chip rendered `:2809`
  (`this.chip(r.src==='real'?'REAL':'SNAPSHOT', …)`). Rows 2–3 hardcode `src:'real'` (`:2715`,`:2716`).
- **Issue:** The chip renders the literal string "REAL" whenever `co.badge` is *truthy*, discarding the
  badge's actual text. Today both fallback cohorts carry `badge:"REAL OASIS"` / `"REAL ADNI"` (both are
  genuinely real cohorts), so the current render is honest. But the honesty is incidental: if a live
  `/api/investigate` cohort ever returned a `badge` like "SURROGATE"/"SYNTHETIC"/"CAL", this code would
  still stamp a green **REAL** chip. Rows 2 (plasma coverage) and 3 (held-out replication) hardcode
  `src:'real'` unconditionally — the underlying values (`biomarker_coverage[bm]`, `replication.effect`)
  are real engine values, so this is defensible today, but again it is not derived from provenance.
  Note `meta.substrate` does not exist in the data (absent), so there is no substrate-level synthetic
  flag to cross-check against; the substrate is `state.substrate` (`oasis`/`adni`, both real).
- **Severity:** LOW/MEDIUM — latent; current data is safe, no live synthetic cohort is wired.
- **Minimal fix:** gate on badge *content* — `src:(/real/i.test(co.badge||'')?'real':'snapshot')` — or
  render `co.badge` text verbatim instead of the hardcoded "REAL"/"SNAPSHOT" literal, so a non-real
  badge cannot be laundered into a REAL chip.

### F5 — LOW — Discovery figure is global, shown under each survivor's "Driving analysis"
- **Where:** `discoveryFigure` `:2723` reads the top-level `data.discovery` (not per-case/per-substrate);
  rendered under the pinned survivor's Artifacts at `:2813`.
- **Issue:** The same clustering figure appears for every card, substrate, and node, but sits under the
  section "Driving analysis" inside a specific survivor's Artifacts, implying it drove *this* branch.
  It does not vary with the pinned mechanism.
- **Severity:** LOW — honest data (once F1 is fixed), loose framing.
- **Minimal fix:** relabel to "Cohort discovery (global)" or select the substrate-appropriate discovery
  object, so it does not read as branch-specific.

## Confirmed clean (no action)

- **Item 1 — no hardcoded numeric literals rendered as engine output.** Every numeric literal in
  `summaryEl`/`artifactsEl`/`discoveryFigure`/`_reasoningBullets`/`_datasetRows` is a layout/scale
  constant, not a data value:
  - `*100` (`:2704`,`:2715`,`:2734`,`:2755`) — percent / score-out-of-100 scaling.
  - `0.5` (`:2731` `auc!==0.5`) — chance-level sentinel that *suppresses* display of a meaningless AUC.
  - `Math.ceil(cf.total/2)` (`:2762`), `.slice(0,2)`/`.slice(0,3)` (`:2717`,`:2798`), pixel sizes —
    thresholds and layout.
  All rendered numbers (AUC, adj, n, CI, margin, confounds pass/total, anchor effect, cluster n/AUC/
  score/silhouette/k) trace to `heroCase()` fields (`naive_effect`, `leakage_margin`, `tests[]`,
  `translation`, `cohort`) or `data.discovery.*`. PASS (subject to F1's *wrong-object* issue and F2/F3's
  *field-pairing* issues, which are provenance/consistency, not fabrication).

- **Item 3 — RNG clusterPlot / dataTable do NOT leak into Artifacts.** The seeded-RNG
  `clusterPlot` (`:2369`, fabricated "k=3 · silhouette **0.42**" `:2387`, invented clusters
  `Control-like`/`Prodromal`/`AD-like` with hardcoded n=24/22/26 `:2372-2374`) and `dataTable`
  (`:2392`, `this.rng(...)` fake subject rows) are consumed **only** by `brainDataEl` (`:2601-2602`,
  the separate "Brain data" tab). Artifacts' "Driving analysis" uses `discoveryFigure` (`:2813`). The
  separation holds — no fabricated figure reaches Artifacts. *(Context note: the fabricated silhouette
  0.42 in Brain data now contradicts the real 0.063 from `discoveryFigure` across tabs — both labeled
  "unsupervised clustering". Pre-existing, out of this tab's scope, but a cross-tab inconsistency worth
  the owning session's attention.)*

- **Item 4 — honest degradation when `heroCase()` is null.** `summaryEl:2750` and `artifactsEl:2782`
  both `return … 'No investigation loaded yet.'`; `discoveryFigure:2723` returns `null` (renders
  nothing). No fabricated fallback is shown. `heroCase()` (`:1841`) returns `null` on any error and
  the tabs never invent numbers. PASS.

- **Item 5 — uncertainty shown where the data has it.** The Summary AUC chip shows the CI when
  `outcome_ci` is present (`:2761`); the first reasoning bullet shows `margin` + `margin_ci`
  (`:2697`,`:2701`); the confounds chip honestly surfaces "N n/a" when tests are `not_available`
  (`:2763`, ADNI survivor renders "4/5 · 1 n/a"). Cluster rows show `n=` (`:2731`). No available CI/n
  is dropped except as noted in F3 (which is a *mismatch*, not omission). PASS.
